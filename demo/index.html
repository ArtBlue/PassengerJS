Passenger JS

The password generation helper that alleviates your user's frustration!

Passenger JS is a UI/UX password helper tool that allows users to create passwords that meet the minimum requirements of a website or web application with greater ease. Instead of multiple page refreshes to indicate to the user what is still missing from the set of password requirements, multiple direct indicators provide instantaneous feedback as users type, to help them come up with passwords that meet the requirements of a website or application. Alongside the password requirement fulfillment feature, there is also a password show toggle ability to allow users to see what they've already typed in the password field. When users are able to create passwords faster, you have lower bounce rates, higher conversions, and you know the rest...

UX research has shown that there is a notable user frustration with user password creation across a wide range of online applications. This tool helps to alleviate that frustration through numerous proven UX principles.

* Passenger JS gives the developer access determine if the specific instance of Passenger JS has passed all the required checks. This gives the power back to the user to determine what to do with the information. You may want to highlight the password box, refuse form submission with a special message - you're free to determine how to direct the user in the instance the user has not fulfilled all the requirements of the field. Simply make the method call off the instance, like this (for the sake of this demo, check the console to see pass/fail state): 

myPassword2Checker.allPassed()

Features

  • Plug-n-play installation requiring nothing but a password field to be passed in during instantiation
  • Customizable through additional class inclusions
  • Ability to add more than one instance on a page
  • Ability to pass in custom password contract rules
  • No dependencies!

Details

  • Full Documentation (JSDocs): see the index file in the docs folder that came with this bundle.
  • Github Repo
  • Continuous Integration: although the code has automated tests running to verify builds, those tests are skim and do not relay the true state of the build (see Unit Tests for more information).
  • Unit Tests: written in Chai/Mocha with a touch of jsdom for instantiation test in Node JS. For a true world test, unit tests should be run manually using /spec/passenger.spec.html as this library is tighly coupled with the DOM and requires a full browser for best real-world tests. To test locally, you will need to install Node JS, and all the required modules.
